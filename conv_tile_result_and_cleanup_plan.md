## Review Corrections

The smallest correct cleanup target is not "default plus im2col" if performance matters. Im2col is a universal lowering in the mathematical sense: a spatial convolution can be materialized as a 1x1 convolution over `IC * KH * KW`. That makes it a correctness fallback, but it also expands 3x3 input traffic by 9x and gives up direct spatial scheduling. If every spatial conv is lowered this way, the direct strategy becomes unnecessary only because the implementation is accepting a large performance regression.

The useful distinction is:

| Count type | Minimal safe set | Meaning |
| --- | --- | --- |
| Semantic strategy families | `direct_h_split`, `pointwise_oc_h_precision_fallback`, `pointwise_oc_h_layout_fallback`, `dense_spatial_im2col_oc_h`, `depthwise_channel_h`, `group_serial_h` | Names explain why each tile exists and which hardware/layout risk it handles. |
| Implementation primitives | `direct_conv_tile`, `im2col_materialize` | One shared direct-conv executor can run H, OC, group, depthwise-channel, and batch slices; im2col is only a producer for selected dense spatial overflow cases. |

ONNC/NVDLA partial-weight modes also need precise terminology. They do not do software OC tiling and output stitching like `conv_new_clean.py`, but they do stream partial output-kernel groups through CBUF. ONNC's `getBanksForPartialWeights()` sizes only `2 * MAC_ATOMIC_K * C * KH * KW` worth of weights, so this is hardware output-kernel/KG streaming rather than a completely different overflow axis. It is cleaner than manual OC stitching only if RK3588 honors the same CBUF/CACC behavior.

The `pointwise_oc_h_precision_fallback` name should be treated as an observed safe fallback, not a proof of arithmetic precision theory. OC tiling reduces weight/output layout pressure, but it does not reduce the number of input-channel products accumulated for each output channel. If the true failure is accumulation depth, the right replacement is IC-split plus NPU-side sum. If OC tiling fixes current tests, the root cause may instead be register layout, CBUF pressure, output stride, or another RK-specific scheduling issue.

## RKNN Runtime Strings Evidence

`librknnrt.so` from the official RKNN toolkit was downloaded to `ref/rknn/librknnrt.so`. Full strings are saved in `ref/rknn/librknnrt.strings.txt`; the curated tiling subset is saved in `ref/rknn/librknnrt_tiling_strings.txt`.

Important strings:

| Evidence | Meaning |
| --- | --- |
| `RKNPU F2`, `RK3588`, `RKNNRegisterTask_f2`, `RKNPUEmitterF2` | The binary has explicit RK3588/F2 register task and emitter paths. |
| `min_weight_banks > 3 && m_Target == RKNNTarget::RKNPU_F2, do conv with ChannelTile` | Official F2 conv compiler chooses ChannelTile when minimum weight-bank pressure exceeds 3 banks. |
| `Conv min_weight_banks > 3, OutputName : %s` | The threshold is not only a dead enum string; there is a conv diagnostic for it. |
| `CNA feature group0/group1`, `CNA weight group0/group1`, `CNA csc group0/group1` | The runtime directly reasons about CNA feature, weight, and CSC groups. |
| `data reuse`, `weight reuse`, `mc_treat_by_y_tile`, `mc_treat_by_k_tile`, `mc_treat_by_1c_y_tile`, `mc_treat_by_1c_k_tile` | The runtime has explicit reuse and Y/K/1C tiling bookkeeping. |
| `Invalid reuse strategy`, `banks num invalid`, `Feature is too large`, `X tile buffer overflow` | The runtime has CBUF/reuse validation around these choices. |

Interpretation: Rockchip's own F2 path appears closer to a direct ChannelTile/OC-family scheduler with reuse metadata than to "im2col everything" or a pure ONNC replacement. This supports keeping the OC/ChannelTile semantic family in `conv_new_clean.py`. The cleanup should remove duplicated loop bodies and make ChannelTile a first-class direct-conv tile descriptor, not delete it.

This also changes the next experiment. The earlier forced weight-bank sweep is useful but incomplete: it proved some hardware weight reuse below the full-weight bank estimate, then failed on the real `160->320 3x3` overflow. The official string suggests the better experiment is to reproduce F2 ChannelTile behavior for `min_weight_banks > 3`: explicit channel/kernel-group tiling, direct spatial layout, and correct CNA feature/weight/CSC group plus reuse programming.

## `run_conv2d` Problem

`run_conv2d` is hard to clean up because it mixes three concerns in every branch: tiling decision, input/weight materialization, and output unpack/stitching. The active test list exercises six separate branch families: default flat/H, default spatial, dense spatial im2col+OC/H, spatial OC-serial, depthwise channel/H, grouped serial, plus the pointwise weight-reuse special case.

The root cause is RK3588's 12-bank CBUF. NVDLA/OpenDLA reference software can often keep FP16 weights resident with 32 banks, but RK3588 needs extra fallback strategies for dense spatial weight pressure and FP16 `IC >> OC` pointwise precision. The current script solves this pragmatically with OC tiling and im2col, but the duplicated loop bodies make output layout and offset bugs likely.

## Smallest Strategy Sets

This is the cleanup-relevant result: do not pick a different best strategy for each shape. Pick the smallest set of mechanism families that covers the active tests, then implement those with shared tile/executor code.

### Safe Now

This set keeps only mechanisms that `conv_new_clean.py` already exercises, so it is the safest cleanup target before adding new hardware behavior.

| Mechanism family | Shapes | Keep/merge |
| --- | ---: | --- |
| `direct_h_split` | 61 | One direct-conv H tiler for default flat and spatial cases; no separate pointwise weight-reuse chain. |
| `pointwise_oc_h_precision_fallback` | 68 | Keep OC/H tiling only for FP16 IC>>OC numerical-risk pointwise cases until IC-split+DPU-sum is proven. |
| `pointwise_oc_h_layout_fallback` | 22 | Temporary pointwise OC/H path for current layout quirks; should collapse into direct_h_split or precision fallback. |
| `dense_spatial_im2col_oc_h` | 11 | Keep im2col+OC/H only for dense spatial cases that current direct scheduling cannot safely handle. |
| `depthwise_channel_h` | 38 | One depthwise channel/H tiler. |
| `group_serial_h` | 17 | One per-group direct/H tiler. |

Safe-now minimal set size: 6 mechanism families. This is already smaller than the current code because `current_default_flat`, `current_default_flat_h`, `current_default_spatial`, `current_pointwise_weight_reuse_h`, and non-overflow `current_spatial_oc_serial` can all be represented as `direct_h_split` tiles executed by one shared loop.

### Cleaner Future

This set is smaller, but it depends on proving two new hardware paths on RK3588: RKNN-style direct ChannelTile for dense spatial overflow and NPU-side sum for IC-split.

| Mechanism family | Shapes | Requirement |
| --- | ---: | --- |
| `direct_h_split` | 87 | Shared direct row tiler for pointwise, normal spatial, small grouped/depthwise-independent cases. |
| `rknn_channel_tile_h` | 7 | Prove direct spatial ChannelTile for dense spatial weight overflow, including output offsets, bank allocation, and reuse flags. |
| `ic_split_sum_h` | 68 | Prove DPU/NPU sum path for partial input-channel convolution outputs; no CPU/GPU accumulation. |
| `depthwise_channel_h` | 38 | Still useful because depthwise has channel layout constraints and tiny per-channel weights. |
| `group_serial_h` | 17 | Still useful because grouped conv is naturally multiple independent convs. |

Cleaner-future minimal set size: 5 mechanism families. If depthwise channel tiling can be expressed as the same generic tile executor with only a channel-slice dimension, the executor code can be shared even though the strategy family remains separate.

## Can It Be Fewer Than 4?

Yes, if we count implementation primitives instead of semantic strategy families. The six safe-now families above collapse to two executor primitives because group, depthwise, OC, channel, and H tiling are all just different slices of a direct conv tile.

### Safe-Now Primitive Set

| Primitive | Shapes using it | What it covers |
| --- | ---: | --- |
| `direct_conv_tile` | 217 | Direct conv executor with optional H, OC, channel, or group slicing. Covers pointwise, direct spatial, grouped serial, depthwise channel tiles, and pointwise OC fallbacks. |
| `im2col_materialize` | 11 | Prepares dense spatial overflow tiles as 1x1 inputs before `direct_conv_tile`. |

Safe-now lower-level set size: 2 primitives. This is possible without new hardware behavior, but it is honest only if `direct_conv_tile` accepts tile descriptors with independent `n`, `group`, `channel`, `oc`, and `h` slices. Then depthwise/grouped/OC/H are descriptor dimensions, not separate loop bodies.

This does not mean "use im2col for everything." It means the executor is shared. The planner should still prefer direct conv tiles and emit `im2col_materialize` only for the dense spatial shapes that the current direct scheduler cannot safely run.

### Cleaner-Future Primitive Set

| Primitive | Shapes using it | Requirement |
| --- | ---: | --- |
| `direct_conv_tile` | 210 | Same shared executor for full-weight direct tiles and RKNN-style ChannelTile descriptors. |
| `rknn_channel_tile` | 7 | Prove direct spatial ChannelTile for dense overflow without im2col. |
| `npu_sum_reduce` | 68 | Prove NPU/DPU sum for IC-split partial outputs. |

Cleaner-future lower-level set size: 3 primitives. It removes im2col materialization, but adds a ChannelTile mode and NPU reduction. This is still fewer than four, but it is less safe until those hardware experiments pass.

### Lower Bound

One primitive is not realistic for a clean implementation. Either safe-now needs `im2col_materialize` for dense spatial overflow, or cleaner-future needs `rknn_channel_tile` plus `npu_sum_reduce` for dense spatial overflow and FP16 IC-split precision. Calling all of those plain `direct_conv_tile` without tile metadata would hide real dataflow differences and make the code less clear, not cleaner.

Using im2col as the only primitive is technically possible but should be rejected as a cleanup goal. It removes strategy complexity by turning every spatial convolution into a larger pointwise convolution, which increases DMA traffic, task count, and CBUF pressure. That is a fallback, not the main path.

## Output Format Finding

`conv_new_clean.py` has three unpackers, but for the active shape list only two are live in the current dispatch: `flat_1x1` and `nc1hwc2`. `grouped_spatial` is still present in the default path, but grouped spatial shapes are intercepted by `current_grouped_serial` before that path, so it is dead for these tests.

A single output format is technically possible only by lowering every conv to 1x1/im2col-style flat output, including grouped and depthwise convs. That would simplify unpacking but is not the best cleanup: it explodes tasks and input materialization for depthwise/grouped/spatial layers and gives up direct-conv hardware scheduling. The practical cleanup target is two output formats: keep `flat_1x1` for pointwise/im2col and `nc1hwc2` for direct spatial/depthwise/group-serial. Delete or quarantine `grouped_spatial` only after confirming no caller reaches it.

## Strategies To Use

- Safe cleanup set: `direct_h_split`, `pointwise_oc_h_precision_fallback`, `dense_spatial_im2col_oc_h`, `depthwise_channel_h`, `group_serial_h`, plus a temporary `pointwise_oc_h_layout_fallback` bucket until those layout cases are proven direct.
- Future cleanup set: `direct_h_split`, `rknn_channel_tile_h`, `ic_split_sum_h`, `depthwise_channel_h`, and `group_serial_h`. ONNC-style `hw_weight_tile_h` remains a separate research path, but the RKNN runtime strings make F2 ChannelTile the more relevant next target.
- If counting implementation primitives, less than four is possible: safe-now is 2 primitives (`direct_conv_tile`, `im2col_materialize`), cleaner-future is 3 primitives (`direct_conv_tile`, `rknn_channel_tile`, `npu_sum_reduce`).
- If counting semantic strategy families, keep the six safe-now names because they explain why each tile exists. The implementation should still share one direct tile executor.
- Do not use CPU/GPU fallback for IC-split accumulation. The sum path must be NPU/DPU if it replaces OC tiling.
- Do not promote ONNC or RKNN-derived strategies from "future" to "safe" until the hardware experiments below pass on RK3588.

## Hardware Experiments Needed

These experiments prove the unverified assumptions required to replace the current pragmatic fallbacks with direct hardware mechanisms. After the RKNN runtime string pass, the dense-spatial experiment should target RKNN-style F2 ChannelTile first, not pure ONNC-style hidden weight streaming. Experiments should be standalone scripts outside `examples/kernel_6_18/` unless that directory is explicitly targeted. The runtime datapath must stay on the NPU; CPU-side NumPy may be used only to build deterministic input/weights and compare final buffers, as existing examples do.

### Experiment 1: RKNN F2 ChannelTile

Goal: prove RK3588 can run a direct dense spatial convolution whose full weights do not fit in 12 CBUF banks by using RKNN-style ChannelTile, without lowering to im2col. This is the experiment directly suggested by the official runtime string: `min_weight_banks > 3 && RKNPU_F2 -> ChannelTile`.

Candidate file: `experimental/prove_conv_hw_weight_stream.py`.

Shapes:

| Shape | Why |
| --- | --- |
| `b1_c32_h14_w14_oc128_wic32_k3x3_g1` | Small enough to debug; choose/tune OC so `min_weight_banks > 3` and ChannelTile should trigger by the RKNN threshold. |
| `b1_c160_h14_w14_oc320_wic160_k3x3_g1` | Real dense spatial overflow case; full FP16 weights are about 900 KiB, far above RK3588 CBUF. |
| `b1_c3_h320_w320_oc32_wic3_k3x3_g1` | Large input/output pressure case; proves row tiling can compose with the same task executor even when weights are small. |

Method:

1. Start from `conv_new_clean.py` direct spatial register generation, not im2col.
2. Compute the same `min_weight_banks` quantity used by the planner as closely as possible, and select at least one shape just below/equal/above the F2 threshold of 3 banks.
3. Split output channels/kernel groups explicitly. For each tile, pack only that tile's weights, program the output-channel offset/surface stride so the final output is contiguous, and preserve direct spatial `nc1hwc2` layout.
4. Vary only one register family at a time: weight bank allocation, weight offset/KG start, output offset, then data/weight reuse flags. Log the register diff for each variant.
5. Compare three outputs for the small shape: current known-good path, candidate ChannelTile path, and NumPy reference. For large shapes, compare candidate against current known-good fallback first, then NumPy on a reduced deterministic subcase if runtime is too high.
6. Add controls with distinct per-OC-tile weight patterns and nonzero output prefill. If a tile writes the wrong channel range or leaves old data, the failure is an output offset/surface issue, not arithmetic.

Pass criteria:

- Max error matches the existing `conv_new_clean.py` tolerance envelope for the same shape.
- Output layout matches `nc1hwc2` without CPU/GPU post-run stitching.
- The same script passes after running `python examples/simple_add.py` before and after it, proving the NPU state was not left polluted.
- Register traces show the path did not use `spatial_im2col` and did not use CPU/GPU accumulation.

Failure interpretation:

| Failure | Likely meaning |
| --- | --- |
| Only the first OC tile is correct | Output-channel offset, weight KG start, or CSC group programming is wrong. |
| Tiles are individually correct but final buffer has gaps/interleaving | `nc1hwc2` destination stride/surface math is wrong. |
| Works just below/equal 3 banks but fails above 3 banks | The experiment is still not reproducing RKNN's ChannelTile threshold behavior. |
| Works for small shape but fails for `160->320` | Bank allotment, KG indexing, or reuse flags are wrong at large KG count. |
| `simple_add.py` fails after the experiment | The register sequence polluted NPU state and must be isolated/reset before more conv tests. |

### Experiment 1B: ONNC/NVDLA Partial-Weight Streaming

Goal: separately test whether RK3588 can do ONNC/NVDLA-style partial-weight streaming with CACC accumulation and no output-channel tiling. This is still interesting, but it is no longer the primary cleanup experiment because official F2 strings point to ChannelTile for `min_weight_banks > 3`.

Use the existing calibrated forced-bank sweep as the starting point. The pass condition is stricter than the current result: the real `b1_c160_h14_w14_oc320_wic160_k3x3_g1` overflow must pass direct spatial without im2col and without software output stitching.

Current proof boundary:

- Candidate file: `experimental/prove_conv_hw_weight_stream.py`.
- The `weight-stream` mode only forces the visible `CNA_CBUF_CON0` data/weight bank split with `patch_cbuf_con0()`. It does not program KG start, tile-table reuse, or CNA feature/weight/CSC group state.
- The calibrated bank sweep shape is `bank_sweep_c16_h16_w16_oc8_wic16_k12x12_g1`. Its full-weight estimate is 3 banks; forced weight banks down to 2 pass, and 1 bank fails. This proves some hardware tolerance/reuse exists below the full estimate.
- The real dense overflow shape is `b1_c160_h14_w14_oc320_wic160_k3x3_g1`. Existing experiments show the current direct registers still fail this shape; even naive 16-wide direct ChannelTile is wrong before final placement.
- The `channel-tile` mode now separately tests OC slicing, output final-buffer placement, PC patching style, and visible `CNA_CBUF_CON0_WEIGHT_REUSE` modes. Those experiments rule out plain PC patching and the visible weight-reuse bit as standalone fixes for the changing-weight chain.
- The `dump-channel-tile-regs` mode shows adjacent 16-wide OC tiles only change weight address, output address, and optionally the visible reuse bit. That state delta is too small to explain RKNN's advertised F2 ChannelTile behavior.

Remaining weight-streaming blockers:

- Exact KG/weight-start register or task field is unknown.
- CACC persistence across partial weight submissions is not proven for changing weight slices.
- Bank-boundary rules for RK3588 F2 are not known beyond the small forced-bank sweep.
- The relation between ONNC/NVDLA partial-weight streaming and RKNN F2 ChannelTile is not 1:1; RKNN's own runtime string points to ChannelTile at `min_weight_banks > 3`.

Practical refactor impact: do not replace `dense_spatial_im2col_oc_h` with a generic `hw_weight_tile_h` path yet. The only safe dense-overflow cleanup is to keep im2col as a producer and factor the execution loop around it.

### Experiment 2: IC-Split Plus NPU Sum

Goal: prove the replacement for `pointwise_oc_h_precision_fallback`: split a pointwise conv along input channels, run multiple NPU convs, and sum partial outputs on NPU/DPU, with no CPU/GPU accumulation.

Candidate file: `experimental/prove_conv_ic_split_npu_sum.py`.

Shapes:

| Shape | Split |
| --- | --- |

2. Run each partial conv as a normal pointwise NPU task that writes a partial output buffer.

4. Compare against current `pointwise_oc_h` output and NumPy reference.
5. Include a control case that uses only the first IC chunk; the final output must differ from the full result, proving the reduction path is active.

Pass criteria:

- Final output matches current known-good output within tolerance.
- Partial outputs are not read back and accumulated by CPU.
- The number of conv tasks equals `ceil(IC / 128)` times the H tiling count, followed by NPU-side sum tasks.

| Failure | Likely meaning |
| --- | --- |
| Each partial conv matches but sum fails | DPU/EW add register path is wrong. |
| Sum works for 2 chunks but not many chunks | Reduction chaining or output reuse state is wrong. |
| Result still has the old precision/layout failure | The original pointwise issue is not solved by IC depth reduction, so the root cause is not just accumulation depth. |

- The `0x1178` vector receives per-tile modulo/start indexes. In the generic path it stores `tile_index % tiling_method`; in the method-`2` path it stores `tile_index & 1`; in method-`3` it stores `tile_index % 3`.
- The `0x1180` vector receives per-tile marker/type flags. The observed values are `1` for normal tiles, `3` for a method-`2` boundary marker, and `7` for method-`3`/generic remainder boundary markers.
- When this tiling-method value is neither `2` nor `3`, the block computes remainders modulo that value and writes one vector with remainder indices and a second vector with marker `1`.
- For tiling method `3`, the block uses marker `7` at a boundary case and otherwise marker `1`.
- For tiling method `2`, the sibling path writes alternating/parity-style values in one vector and markers `1` or `3` in the other vector.
- The repeated structure appears twice: once for a `w20` loop and once for a `w19` loop, likely two dimensions or two split vectors rather than two separate high-level strategies.
- Therefore `2` and `3` are confirmed tiling-method cases inside `fcn.005f38f0`, and `0x1178`/`0x1180` are now narrowed to paired per-tile index/type vectors. Their semantic names are still unknown. Do not map them to `Y`, `K`, `1C`, or ChannelTile yet without naming the object fields that choose `[x29+0x1188]`.

Practical refactor impact: do not encode these constants into `conv_new_clean.py` as strategy enums. They are tile-table composer values, not hardware register values. The safe Python cleanup should keep explicit strategy names until the tiling-method source feeding `[x29+0x1188]` is named and the later tile-table-to-register updater maps the `0x1178`/`0x1180` vectors to concrete CNA/DPU/PC register entries.

Known local PC-chain mechanics:

- `write_regs_to_npu_task()` appends a four-QWORD tail to each task: `PC_BASE_ADDRESS`, `PC_REGISTER_AMOUNTS`, `VERSION`, then `PC.OPERATION_ENABLE`.
- The next task address is `regcmd_dma + next_offset * sizeof(uint64)` masked with `0xFFFFFFF0`.
- The common Python chain fetch count is `_ceil_div(next_task_regs_len, 2) + 1`; Mesa-style raw code instead patches `regs_to_fetch = align((next_size - 4) // 2, 2)`.
- H-split flat output placement uses `output_offset = row_start * out_w * 16` in the newer Python examples.
- OC-tiled flat placement uses `surface_offset = (oc_start // 16) * out_width_stride * 16 * bytes_per_element`.
- Mesa-style raw spatial placement uses `task.output_offset = calc_line_stride(output_width) * output_height_processed`.
- `weight_reuse` only sets `CNA_CBUF_CON0_WEIGHT_REUSE` for later tasks in the H-chain path; OC-tile paths with different `weight_dma + weight_offset` do not currently have a proven reuse sequence.

PC-chain reverse update from local `conv*.py` files:

- The safe H-chain model is now concrete: each task copies a separate input tile at `input_dma + input_offset`, writes to `output_dma + row_start * out_w * 16`, keeps the same `weight_dma`, and sets `weight_reuse=bool(task_regs)` after the first task.
- The older OC-chain model is also concrete but not proven for the current target: each OC tile uses `weight_dma + oc_start * KH * KW * aligned_in_c * bytes` and `output_dma + row_offset + (oc_start // 16) * out_width_stride * 16 * bytes`. These paths often also force full data bank and tile data banks, but they do not establish RKNN's data/weight reuse table or CNA group setup.
- Mesa-style PC patching differs from the helper used in `conv_new_clean.py`: Mesa aligns each regcmd block to 64 bytes and writes `PC_REGISTER_AMOUNTS = align((next_size - 4) // 2, 2)`, while the Python helper uses QWORD offsets and `_ceil_div(next_task_regs_len, 2) + 1`. The small-proxy ChannelTile A/B failed identically with both patchers, so this is no longer the leading cause.
- Kernel-6.18 examples have an additional task-count distinction: `examples/kernel_6_18/conv_new_clean.py:631-676` and `examples/kernel_6_18/simple_conv_fp16.py` write the PC tail into the regcmd BO and set `regcfg_amount = len(regs) + len(tails)`, while older `examples/conv.py:594-645` and `examples/conv_gemm.py:742-799` keep `regcfg_amount = len(regs)`. This difference matters for normal kernel submission hygiene, and a shared executor must preserve the target file's tail-counting convention. It is not enough to explain the changing-weight ChannelTile failure because the current `conv_new_clean.py`/kernel-6.18 path already uses the tail-counting convention.
- The visible `CNA_CBUF_CON0_WEIGHT_REUSE` bit is also not sufficient: never setting it, setting it after the first task, and setting it for every task all fail identically on the small-proxy chained ChannelTile case.
- Output offset is probably not the first failure for dense ChannelTile: the small proxy passes with per-submit final-buffer writes, but the real `160->320 3x3` fails before placement. That points back to weight/KG indexing, CBUF bank allocation, or reuse/group fields.
